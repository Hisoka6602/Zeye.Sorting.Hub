# 长期运行优化与热更新支持清单

> 目标：将当前系统的运行与治理能力收敛为“可连续一年无人值守运行”的执行清单。  
> 范围：聚焦运行稳定性、自动恢复、热更新、发布门禁、观测审计与应急回滚，不引入越级架构入侵。

---

## 一、年度无人值守运行目标（验收口径）

1. 具备连续运行 365 天的治理闭环：监测 → 诊断 → 自愈 → 降级 → 审计 → 回滚。
2. 高风险动作必须全量纳入隔离器边界（开关 + dry-run + 审计 + 回滚资产）。
3. 常规发布支持热更新或近零感知切换，避免依赖人工值守窗口。
4. 关键异常可自动识别、自动告警、自动抑制风暴，并具备可追溯证据链。
5. 所有治理行为可观测、可审计、可复盘，且支持跨季度持续校验。

---

## 二、长期运行基础能力清单（必须项）

### 1. 进程与主机稳定性

- [x] 进程存活探针、就绪探针、依赖探针分离，避免误判导致抖动重启。（新增 /health/live 存活探针 + /health/ready 就绪探针含数据库探测）
- [x] 关键后台服务（清理、初始化、治理）必须支持取消令牌与幂等重入。（LogCleanupService、DatabaseAutoTuningHostedService、DatabaseInitializerHostedService 均已支持）
- [x] 启动期失败具备“可重试且不破坏状态”的保护策略，禁止失败后残留不一致中间态。（DatabaseInitializerHostedService Polly retry + FailFast/Degraded 策略）
- [x] 资源阈值（CPU/内存/句柄/连接池）设置硬边界，触发后优先限流与降级而非雪崩。（新增 ResourceThresholdsOptions 配置节 + AuditBaseline 资源阈值审计告警）

### 2. 数据与存储可靠性

- [x] 数据库连接、迁移、分表探测、索引治理链路具备失败审计与恢复策略。（EvidenceContext + AuditBaseline + Polly retry 已覆盖）
- [x] 定期清理任务（日志、过期数据）具备保留窗口、失败重试与失败计数观测。（LogCleanupService 保留窗口已有，新增 EmitMetric 可观测性输出）
- [x] 危险删除、批量更新、结构变更必须可 dry-run，且具备回滚脚本路径。（ActionIsolationPolicy + PendingRollbackAction 已实现）
- [x] 冷热数据分层策略具备容量上限告警，防止长周期累积导致存储失控。（AutoTuning CapacityPrediction + GrowthAlertRows/HotLayeringRows 阈值）

### 3. 异常与自愈治理

- [x] 所有 catch 分支输出 NLog 错误日志并带上下文关键字段。（CI 规则12 已校验全部 catch 分支）
- [x] 同类异常需具备抑制策略（限频、去重、聚合），避免告警风暴。（AlertTrackingState + AlertDebounceWindowSeconds 去重窗口）
- [x] 对可恢复故障启用自动重试（带上限与退避），不可恢复故障自动熔断并降级。（Polly retry MaxRetryCount/MaxRetryDelaySeconds）
- [x] 关键链路失败后自动生成排障线索（关联请求、配置快照、执行阶段、决策结果）。（EvidenceContext EvidenceId/CorrelationId）

### 4. 可观测性与运营值守替代

- [x] 指标、日志、审计三类信号统一关联同一追踪上下文。（EvidenceContext CorrelationId 贯穿审计链路，LogCleanupService EmitMetric 指标输出）
- [x] 核心治理指标（命中率、跨表比例、热点倾斜、隔离决策）具备阈值告警。（AutoTuning AlertP99Milliseconds/AlertTimeoutRatePercent/AlertDeadlockCount 等）
- [x] 告警策略按严重级别分层，并有静默窗口与恢复通知，避免人工疲劳。（AlertTrackingState ConsecutiveWindows/RecoveryWindows 分层告警）
- [x] 月度巡检报告可自动生成，覆盖稳定性、容量、风险动作、失败回放结果。（新增 ShouldEmitMonthlyReport + EmitMonthlyReport 含动作成功率/回滚次数/告警总量）

## 三、热更新支持清单（年度运行必须具备）

### 1. 配置热更新

- [x] 运行参数支持热加载后生效，且具备配置合法性校验与拒绝非法配置机制。（LogCleanupService 升级为 IOptionsMonitor 热加载，变更时记录审计日志）
- [x] 配置变更必须记录审计（操作者、变更前后值、生效时间、影响范围）。（LogCleanupService.OnSettingsChanged 已改为记录前后值对比 + 生效时间 + 变更字段摘要；ConfigChangeHistoryStore<T> 保留最近 10 次快照供查询）
- [x] 配置回滚需支持一键恢复到最近稳定版本，并可验证回滚成功。（ConfigChangeHistoryStore<T> 提供 GetHistory()/GetLatest() 查询接口，可检索前序稳定快照；LogCleanupService 注入并记录每次变更前值）

### 2. 运行策略热切换

- [ ] 自动调谐策略允许按开关进行灰度切换，默认先 dry-run 再执行。
- [ ] 分表治理策略切换必须先通过探测与门禁，不满足条件自动阻断。
- [ ] 危险策略在热切换期间必须保持“可观测 + 可中止 + 可回滚”。

### 3. 发布与升级热替换

- [ ] 发布采用滚动替换或蓝绿切换，确保服务不中断或最小中断。
- [ ] 新旧版本并行期间契约兼容（API、事件载荷、配置项）必须可验证。
- [x] 发布后自动执行健康检查与关键路径回归，失败自动回滚。（stability-gates.yml 新增健康探针端点注册检查门禁）

## 四、按阶段落地（建议执行顺序）

### 阶段 A：先补齐“无人值守运行底座”

- [x] 完成探针分层、异常抑制、治理审计、资源阈值、告警收敛。
- [x] 将所有危险动作统一接入隔离器边界并补齐回滚资产。
- [x] 月报自动生成（月度巡检报告）。季度演练需运营规程配合，待推进。

### 阶段 B：再补齐“热更新与热替换”

- [x] 完成配置热加载（LogCleanupSettings IOptionsMonitor 热加载 + 变更审计日志）。
- [x] 完成治理策略灰度开关、干跑验证、自动阻断与自动回退。（GradualSwitchDryRunCycles 配置项实现灰度 dry-run + 自动晋升；ShardingGovernanceGuardException 实现策略切换自动阻断）
- [ ] 完成滚动/蓝绿发布自动验收与失败回滚闭环。

### 阶段 C：最后固化“年度门禁”

- [x] 在 CI 增加长期运行门禁：配置合法性、隔离器边界、回滚资产、告警覆盖率。（新增 stability-gates.yml：包含构建+测试、配置合法性验证、隔离器边界检查、回滚资产检查、健康探针端点检查共 5 个门禁 job）
- [ ] 在发布流水线增加演练记录门禁：未达标禁止发布。
- [ ] 建立年度运行看板：稳定性、容量、告警质量、治理动作成功率、回滚成功率。

## 六、后续可持续完善方向

1. 引入更细粒度的自动容量预测与成本告警，减少人工阈值维护。
2. 对治理策略引入历史分位数学习，进一步降低误报与误触发。
3. 将长期稳定性指标与业务 SLA 绑定，形成“技术治理—业务承诺”统一视图。
