# Zeye.Sorting.Hub 模块能力地图

> 目的：用一份持续维护的能力地图替代已完结的阶段性计划文档，帮助后续开发快速判断“当前系统已经能做什么、哪些模块已落地、哪些模块仍是预留”。

---

## 1. 项目定位

`Zeye.Sorting.Hub` 当前更接近“分拣数据中枢 / 数据库治理底座”，而不是完整的现场执行系统。

当前已明确具备：

- Parcel 数据接入、查询、治理与长期存储能力
- 分表、归档、备份、迁移、保留策略、慢查询治理等数据库底座能力
- 面向长期运行的健康探针、后台治理任务、门禁脚本、运行手册与验收资料

当前明确不做或尚未落地：

- 硬件控制、PLC/IO、现场实时执行状态机
- 完整鉴权体系
- UI
- 完整的 Realtime / RuleEngine / Analytics 子域实现

## 3. 对外能力地图

### 3.1 Parcel 查询与管理

- 只读查询：分页、游标分页、详情、邻近记录
- 管理写接口：新增、批量缓冲写入、状态更新、删除
- 治理接口：过期数据清理

主要入口：

- `Routing/ParcelReadOnlyApiRouteExtensions.cs`
- `Routing/ParcelAdminApiRouteExtensions.cs`

### 3.2 数据治理

- 归档 dry-run 任务创建、查询、重试
- Outbox 消息追加与分页查询
- 数据保留策略与自动清理
- 备份、恢复校验、演练资料

主要入口：

- `Routing/DataGovernanceApiRouteExtensions.cs`
- `Infrastructure/Persistence/Archiving/`
- `Infrastructure/Persistence/Retention/`
- `Infrastructure/Persistence/Backup/`

### 3.3 诊断与观测

- `/health/live`、`/health/ready`
- 慢查询画像列表与详情
- 数据库连接诊断与预热
- 查询模板治理、索引建议、自动调谐观测

主要入口：

- `Program.cs`
- `Routing/DiagnosticsApiRouteExtensions.cs`
- `Infrastructure/Persistence/Diagnostics/`
- `Infrastructure/Persistence/AutoTuning/`
- `Infrastructure/Persistence/QueryGovernance/`

### 3.4 审计

- Web 请求审计写入中间件
- 审计只读接口已实现，但默认按配置关闭

主要入口：

- `Middleware/WebRequestAuditLogMiddleware.cs`
- `Routing/AuditReadOnlyApiRouteExtensions.cs`

## 4. 长期运行能力地图

当前系统已经具备以下“底座型”长期运行能力：

- 后台任务：日志清理、数据库初始化、连接预热、缓冲刷盘、分表预建、分表巡检、归档、备份、数据保留、查询治理、Outbox 分发、自动调谐
- 门禁体系：数据库底座门禁、稳定性门禁、EF 迁移校验、压测资产 smoke 校验、Copilot 规则校验
- 运行资料：生产 Runbook、数据库故障应急预案、分表治理 Runbook、备份恢复演练 Runbook、无人值守运行检查清单

---

## 5. 当前能力边界

可以视为“已经成熟”的部分：

- 以 Parcel 为核心的数据接入、查询与治理主链路
- 长期数据库底座路线中的 PR-A ～ PR-T 主体能力
- EF Core 9、多 Provider、分表与迁移治理体系

仍应视为“下一阶段”的部分：

- Realtime、RuleEngine、Analytics 子域
- 完整鉴权与权限策略
- 真正落地的实时业务协同能力
- 完整压测结果的持续回填与运营化沉淀
