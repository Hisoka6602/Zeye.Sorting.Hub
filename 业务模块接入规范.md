# 业务模块接入规范

## 1. 适用范围

本规范用于约束 `Zeye.Sorting.Hub` 在长期数据库底座完成后新增业务模块时的目录结构、命名边界、接口职责与运行期治理接入方式，避免后续模块继续出现随意放文件、随意查库、重复造轮子的情况。

## 2. 强制目录结构

每个新增业务模块必须遵循以下最小结构：

```text
Domain
├── Aggregates/{ModuleName}
├── Enums/{ModuleName}
├── Events/{ModuleName}
└── Repositories/I{ModuleName}Repository.cs

Application
└── Services/{ModuleName}

Contracts
└── Models/{ModuleName}

Infrastructure
├── Repositories/{ModuleName}Repository.cs
└── EntityConfigurations/{ModuleName}EntityTypeConfiguration.cs

Host
└── Routing/{ModuleName}ApiRouteExtensions.cs

Host.Tests
└── {ModuleName}ApiTests.cs
```

## 3. 分层接入要求

### 3.1 Domain

- 只放聚合、实体、值对象、领域事件、领域枚举、领域仓储契约。
- 禁止直接出现 EF Core、SQL、HTTP、Swagger、配置绑定、HostedService。
- 事件载荷必须定义在 `Events` 子目录下，并使用 `readonly record struct`。

### 3.2 Application

- 只放用例编排、参数校验、合同映射、跨仓储流程协调。
- 写命令优先输出统一结果模型，失败时必须带稳定错误码。
- 需要站点 / 产线 / 设备 / 工作站边界时，必须复用 `OperationalScopeNormalizer`、`OperationalScopeRequest` 与 `OperationalScopeResponse`。

### 3.3 Contracts

- 只放请求合同、响应合同、分页合同、对外枚举。
- 禁止直接返回领域实体。
- 禁止新增业务私有的影分身分页模型与影分身结果模型。

### 3.4 Infrastructure

- 负责仓储实现、EF Core 配置、数据库查询、只读模型与持久化治理。
- 禁止把基础设施细节泄漏到 Application 或 Domain。
- 需要批量写入时优先复用现有 WriteBuffer 能力，不得复制新的队列实现。

### 3.5 Host

- 只放路由、依赖注入接线、健康检查、后台服务注册与 Swagger 注释。
- 路由必须通过独立扩展类注册，不得把实现继续堆到 `Program.cs`。
- 路由约定必须优先复用 `EndpointRouteBuilderConventionExtensions`。

## 4. 查询与写入治理要求

### 4.1 查询

- 业务查询必须有时间范围保护，默认按本地时间语义解析。
- 高频列表必须优先游标分页；只有明确需要跳页时才允许页码分页。
- 报表类查询必须复用只读副本与预算守卫，不得直接放开大范围明细查询。
- 查询返回合同不得暴露 EF 实体、`IQueryable` 或数据库 Provider 细节。

### 4.2 写入

- 写入必须考虑幂等；重复请求必须优先复用 `IdempotencyGuardService`。
- 批量写入必须优先复用 `IBufferedWriteService`，禁止新增影分身后台写队列。
- 涉及危险动作（批量删除、批量更新、DDL、恢复）必须走隔离器、dry-run、审计与回滚资产。

### 4.3 事件

- 需要业务事件持久化时必须优先使用 Outbox。
- 需要消费外部事件时必须优先使用 Inbox。
- 禁止绕过现有 Outbox / Inbox 底座自行实现事件状态表。

## 5. API 结果与错误处理要求

- API 必须输出中文摘要、中文描述、中文错误消息。
- 应用层失败结果必须使用稳定错误码，不得让调用方依赖异常消息文本。
- Host 层失败响应必须复用统一 ProblemDetails 映射策略。
- 不允许新增绕过全局异常出口的业务私有错误协议。

## 6. Copilot 落地检查清单

新增业务模块前，必须逐项确认：

1. 已核对当前长期数据库底座台账，并明确从哪个 PR 断点继续。
2. 已明确模块目录结构与命名，不会破坏 DDD 分层边界。
3. 已确认查询路径是否需要时间范围保护、游标分页、只读副本预算守卫。
4. 已确认写路径是否需要幂等、WriteBuffer、Outbox、Inbox。
5. 已确认错误结果是否复用 `ApplicationResult`、`ApplicationErrorCodes` 与统一 ProblemDetails 映射。
6. 已确认新增 / 删除文件会同步更新 `README.md`、`更新记录.md`、`检查台账/文件清单基线.txt` 与对应 PR 台账。

## 7. 当前阶段禁止事项

- 禁止在业务模块中引入 UTC 时间语义。
- 禁止直接在 Host 层访问仓储实现。
- 禁止新增 `Manager`、`Helper`、`Wrapper`、`Facade` 一类的同义胶水类。
- 禁止为了图快复制既有模块后再局部微调。
- 禁止在热路径做反射、多次枚举、同步阻塞数据库调用。
