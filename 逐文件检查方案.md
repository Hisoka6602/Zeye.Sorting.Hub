# 逐文件检查方案（Copilot 执行版）

## 1. 目标与范围

- 目标：逐文件检查仓库，识别并记录以下问题：影分身代码、过度设计、冗余代码、性能问题、逻辑问题、并发竞态问题、违反 `.github/copilot-instructions.md` 规则的问题。
- 范围：仅做检查与记录，不在本方案 PR 内修改业务代码。
- 约束：检查必须可追踪、可分批（多 PR）推进、可回溯，且确保无遗漏。

## 2. 分批执行策略（多 PR）

按目录分批提交“检查结果 PR”，每个 PR 只做“检查结论 + 证据 + 后续修复建议”，不混入无关内容：

1. PR-A：根目录 + `.github/` + 文档与脚本
2. PR-B：`Zeye.Sorting.Hub.Domain/`
3. PR-C：`Zeye.Sorting.Hub.Application/` + `Zeye.Sorting.Hub.Contracts/`
4. PR-D：`Zeye.Sorting.Hub.Infrastructure/`
5. PR-E：`Zeye.Sorting.Hub.Host/`
6. PR-F：`Zeye.Sorting.Hub.SharedKernel/` + `Zeye.Sorting.Hub.Host.Tests/` + 占位子域项目

> 若单个目录文件数过多，可继续按子目录拆分为 PR-B1、PR-B2 等，原则是不遗漏文件。

## 3. 防遗漏机制（必须执行）

### 3.1 生成“检查清单基线”

先导出全仓库文件清单（排除 `bin/`、`obj/`、`.git/`），作为唯一基线。后续每个 PR 都以此基线为核对来源。

### 3.2 逐文件打勾

每检查完一个文件，立刻在“逐文件检查台账”中打勾并记录结果（通过/问题）。

### 3.3 PR 结束前双重对账

1. 对账“本 PR 计划检查文件集合”是否全部有记录。  
2. 对账“仓库基线文件总数”与“累计已检查文件总数”差异，差异必须为 0 才能宣布全量完成。

## 4. 单文件检查项（统一模板）

对每个文件都执行以下检查项：

1. 影分身代码：是否存在复制粘贴同义实现、近似重复逻辑未抽取。
2. 过度设计：是否出现无必要抽象、无实际价值层级、空转封装。
3. 冗余代码：死代码、未使用参数/字段/方法、重复分支。
4. 性能问题：热路径分配、重复计算、低效集合操作、阻塞调用。
5. 逻辑问题：边界条件、空值处理、错误分支、状态流转一致性。
6. 并发竞态：共享状态读写安全、锁粒度、异步并发时序风险。
7. 规则合规：是否违反 `.github/copilot-instructions.md`（时间语义、分层边界、命名、注释、日志、事件载荷、枚举位置等）。
8. 可修复性：是否可在后续小 PR 内低风险修复。

## 5. 问题分级与处置

- P0：会导致错误结果、数据风险、严重并发问题、明确违反硬性规则且影响运行。
- P1：高概率隐患（性能/逻辑/结构），短期应修复。
- P2：规范性与可维护性问题，可排期修复。

处置要求：

- 每个问题必须绑定“文件路径 + 行号区间 + 证据描述 + 分级 + 建议修复 PR”。
- 禁止只给泛化结论，不给定位证据。

## 6. 逐文件检查台账格式（建议）

| 文件路径 | 检查状态 | 问题数(P0/P1/P2) | 主要问题标签 | 证据位置 | 建议修复PR |
|---|---|---|---|---|---|
| `示例/文件.cs` | 已检查 | 0/1/0 | 冗余代码 | L23-L48 | PR-XXX |

状态值统一：`未检查 / 已检查 / 复核完成`。

## 7. 每个检查 PR 的固定交付内容

1. 本批次覆盖文件列表（与基线映射）。
2. 逐文件检查台账（本批次增量）。
3. 问题清单（按 P0/P1/P2）。
4. 未覆盖文件清单（若有，必须写明原因与下个 PR 计划）。
5. 下一批 PR 计划（目录级）。

## 8. 验收标准（全量完成）

- 基线文件覆盖率 100%。
- 每个文件至少有 1 条检查记录。
- 所有 P0 均有明确修复计划并已立项到后续 PR。
- 无“仅结论无证据”的问题条目。

## 9. 执行提醒

- 本方案 PR 不改业务代码，只交付检查方法与流程。
- 后续每个“检查结果 PR”仍需遵守仓库规则：分层边界、注释规范、命名规范、时间本地语义、NLog 落盘要求等。
