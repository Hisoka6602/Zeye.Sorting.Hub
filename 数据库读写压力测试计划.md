# 数据库读写压力测试计划

> **适用版本**：Zeye.Sorting.Hub（EF Core 9.0.14 + EFCore.Sharding 9.0.10 + MySQL 8.x）  
> **测试目标库**：`zeye_sorting_hub`（MySQL 127.0.0.1:3306）  
> **文档用途**：为包裹分拣系统数据库的写入性能、查询性能、混合读写及长时稳定性提供可量化的压测验收基线，同时验证 AutoTuning 闭环自治能力在高压下的准确性。

---

## 一、测试目标

| 目标维度 | 具体指标 |
|---------|---------|
| **写入吞吐量** | 包裹主表（Parcels）单节点写入 TPS ≥ 500 |
| **查询吞吐量** | 常见索引查询（BarCodeInfos 明细表 / BagCode / ScannedTime 范围）QPS ≥ 800 |
| **端到端延迟** | P95 < 200 ms，P99 < 500 ms（与 `Persistence:AutoTuning:AlertP99Milliseconds = 500` 对齐） |
| **错误率** | 整体错误率 < 1%（与 `Persistence:AutoTuning:AlertTimeoutRatePercent = 1` 对齐） |
| **死锁** | 整个压测周期死锁次数 = 0（与 `Persistence:AutoTuning:AlertDeadlockCount = 1` 对齐） |
| **分表命中率** | `autotuning.sharding.hit_rate` 指标 ≥ 98% |
| **连接池** | 峰值连接数 < `max_connections × 80%`，不出现连接池耗尽 |
| **稳定性** | 持续压测 30 分钟，无 OOM、内存泄漏、连接泄漏 |

## 二、测试环境

### 2.1 推荐最低配置

| 资源 | 压测执行机 | 数据库节点 |
|------|-----------|-----------|
| CPU | 4 核 | 8 核 |
| 内存 | 8 GB | 16 GB |
| 存储 | SSD（临时即可） | NVMe SSD |
| 网络 | 万兆内网互通 | — |

> **说明**：若在单机 Docker Compose 环境执行，数据库容器至少分配 4C/8G，避免容器资源竞争干扰数据。

### 2.2 MySQL 关键参数确认

在压测前执行以下 SQL 确认参数并记录基线：

```sql
-- 确认关键参数
SHOW VARIABLES LIKE 'innodb_buffer_pool_size';
SHOW VARIABLES LIKE 'max_connections';
SHOW VARIABLES LIKE 'innodb_flush_log_at_trx_commit';
SHOW VARIABLES LIKE 'sync_binlog';
SHOW VARIABLES LIKE 'innodb_io_capacity';
SHOW VARIABLES LIKE 'thread_cache_size';
```

**推荐最低值**（供参考，生产环境按实际调整）：

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| `innodb_buffer_pool_size` | 物理内存的 60~70% | InnoDB 缓冲池，影响热数据命中 |
| `max_connections` | ≥ 500 | 应大于 EF Core DbContext 池并发量的 3 倍 |
| `innodb_flush_log_at_trx_commit` | 1（严格）/ 2（权衡） | 1 最安全，2 高吞吐 |
| `sync_binlog` | 1（严格）/ 0（权衡） | Binlog 刷新策略 |

### 2.3 应用侧关键配置（`appsettings.json`）

压测期间应确保以下配置就位，与压测目标保持一致：

```json
{
  "Persistence": {
    "Provider": "MySql",
    "Sharding": {
      "CreateShardingTableOnStarting": true,
      "ParcelRelatedHashShardingMod": 16
    }
  },
  "PerformanceTuning": {
    "CommandTimeoutSeconds": 30,
    "MinCommandElapsedMilliseconds": 50,
    "MaxRetryCount": 5,
    "MaxRetryDelaySeconds": 10
  },
  "AutoTuning": {
    "SlowQueryThresholdMilliseconds": 500,
    "AnalyzeIntervalSeconds": 30,
    "Autonomous": {
      "Execution": {
        "Isolator": {
          "EnableGuard": true,
          "AllowDangerousActionExecution": false,
          "DryRun": true
        }
      }
    }
  }
}
```

> **注意**：压测期间将 `DryRun` 设为 `true`，防止 AutoTuning 在高压期间自动执行 DDL 影响测试结果；压测结束后还原为 `false`。

### 4.2 初始化种子数据（sysbench 方式）

sysbench 不直接操作 EF Core 模型，故使用自定义 Lua 脚本或直接使用 MySQL INSERT 批量写入种子数据，用于读取压测的数据基础：

```sql
-- 快速生成 100 万条测试包裹（Pending 状态，条码唯一）
DELIMITER $$
CREATE PROCEDURE seed_parcels(IN total INT)
BEGIN
  DECLARE i INT DEFAULT 0;
  WHILE i < total DO
    INSERT INTO `Parcels_202603`  -- 替换为实际本月分表名
    (ParcelTimestamp, Type, Status, BarCodes, Weight, BagCode, WorkstationName,
     ScannedTime, DischargeTime, TargetChuteId, ActualChuteId,
     Length, Width, Height, Volume, IsSticking, HasImages, HasVideos,
     Coordinate, NoReadType, RequestStatus, CreatedTime, UpdatedTime)
    VALUES
    (UNIX_TIMESTAMP() * 1000 + i, 0, 0,
     CONCAT('BC', LPAD(i, 12, '0')),
     ROUND(RAND() * 30 + 0.1, 3),
     CONCAT('BAG', LPAD(FLOOR(i / 10), 8, '0')),
     CONCAT('WS', LPAD(FLOOR(RAND() * 20), 3, '0')),
     NOW(), DATE_ADD(NOW(), INTERVAL 3 SECOND),
     FLOOR(RAND() * 200 + 1), FLOOR(RAND() * 200 + 1),
     ROUND(RAND() * 40 + 5, 3), ROUND(RAND() * 30 + 3, 3), ROUND(RAND() * 20 + 2, 3),
     ROUND(RAND() * 24000 + 30, 3),
     FALSE, FALSE, FALSE, '', 0, 0,
     NOW(), NOW());
    SET i = i + 1;
  END WHILE;
END$$
DELIMITER ;

CALL seed_parcels(1000000);
DROP PROCEDURE seed_parcels;
```

> **注意**：种子数据用于查询场景（Scenario 2），写入场景（Scenario 1）不依赖预存数据。

## 五、测试场景设计

### Scenario 1：纯写入压测（模拟包裹入库）

**目的**：测量 Parcels 主表及关联表的最大写入吞吐量，验证分表路由的写入开销。

**执行方式**：使用应用层写入接口（或直接 SQL INSERT，优先通过应用层以包含 EF Core 开销）

> **⚠️ 重要说明：sysbench 原生脚本 vs 业务表压测**
>
> sysbench 内置的 `oltp_write_only.lua`/`oltp_read_only.lua`/`oltp_read_write.lua` 脚本会在独立的 **sbtest\*** 测试表上运行，**不会操作** `Parcels_*`、`BagInfos_*` 等业务分表。其作用是建立 **MySQL 原生 InnoDB 读写基线**（排除业务逻辑变量），用于评估数据库层吞吐天花板与延迟基线。
>
> 若需覆盖分表路由、索引命中、EF Core 开销等业务关键点，须使用**第 4.3 节自定义 SQL 脚本**或通过**应用层写入接口**注入。两类测试互补，建议均执行。

#### 1.1 MySQL 原生写入基线（sysbench sbtest 表）

> **用途**：排除业务逻辑，评估当前 MySQL 实例的原生 InnoDB 写入吞吐上限与延迟水位。

```bash

#### 1.2 业务表写入压测（Parcels_* 分表，应用层接口）

> **用途**：验证包含 EF Core 分表路由开销的真实写入性能，覆盖 Parcels_*/Parcel_BarCodeInfos_*/BagInfos_* 等业务表。

推荐通过应用层 HTTP 接口或直接 SQL INSERT 模拟包裹入库，每条记录插入 Parcels_* 主分表，详见**第四节"测试数据准备 — 种子数据存储过程"**。也可编写自定义 sysbench Lua 脚本直接操作业务分表，示例：

```lua
-- /tmp/parcels_write.lua（自定义业务表写入脚本）
-- 此脚本直接向当月 Parcels_* 分表执行 INSERT，覆盖业务索引路径
local function event()
  local ts = os.time() * 1000 + sysbench.rand.uniform(0, 999)
  local bc = string.format("BC%012d", sysbench.rand.uniform(1, 999999999999))
  local bag = string.format("BAG%08d", sysbench.rand.uniform(1, 99999999))
  db:query(string.format([[
    INSERT INTO `Parcels_202603`
    (ParcelTimestamp, Type, Status, BarCodes, Weight, BagCode, WorkstationName,
     ScannedTime, DischargeTime, TargetChuteId, ActualChuteId,
     Length, Width, Height, Volume, IsSticking, HasImages, HasVideos,
     Coordinate, NoReadType, RequestStatus, CreatedTime, UpdatedTime)
    VALUES (%d, 0, 0, '%s', %.3f, '%s', 'WS001',
     NOW(), DATE_ADD(NOW(), INTERVAL 3 SECOND),
     %d, %d, %.3f, %.3f, %.3f, %.3f,
     0, 0, 0, '', 0, 0, NOW(), NOW())
  ]], ts, bc,
    sysbench.rand.uniform_double() * 30 + 0.1, bag,
    sysbench.rand.uniform(1, 200), sysbench.rand.uniform(1, 200),
    sysbench.rand.uniform_double() * 40 + 5,
    sysbench.rand.uniform_double() * 30 + 3,
    sysbench.rand.uniform_double() * 20 + 2,
    sysbench.rand.uniform_double() * 24000 + 30))
end

sysbench.hooks.report_intermediate = sysbench.report_default


| 阶段 | 并发线程数 | 持续时间 | 预期 TPS |
|------|-----------|---------|---------|
| 热身 | 4 | 60 s | 基线建立 |
| 低压 | 8 | 120 s | ≥ 200 TPS |
| 中压 | 16 | 120 s | ≥ 400 TPS |
| 高压 | 32 | 120 s | ≥ 500 TPS |
| 峰值 | 64 | 60 s | 观察极限，允许 TPS 下降 |
| 冷却 | 8 | 60 s | TPS 应恢复到低压水平 |

**观察指标**：
- TPS（transactions/sec）
- P95/P99 写入延迟
- `SHOW STATUS LIKE 'Innodb_rows_inserted'`（累计插入行数）
- `SHOW PROCESSLIST`（活跃连接数）
- 应用侧 `logs/database-*.log` 中慢查询条目数

**通过标准**：
- 高压阶段（32 线程）TPS ≥ 500
- P99 ≤ 500 ms
- 错误率 = 0%（EF Core 重试机制下最终成功）

### Scenario 4：长时稳定性压测（Soak Test）

**目的**：持续 30 分钟中等负载，验证无连接泄漏、无内存增长、分表路由稳定、AutoTuning 无误触发回滚。

#### 4.1 压测配置

```bash
# 30 分钟稳定性压测（中等负载，读写混合）
sysbench \
  --db-driver=mysql \
  --mysql-host=127.0.0.1 \
  --mysql-port=3306 \
  --mysql-db=zeye_sorting_hub \
  --mysql-user=root \
  --mysql-password=$MYSQL_PWD \
  --table-size=1000000 \
  --tables=4 \
  --threads=16 \
  --time=1800 \
  --report-interval=60 \
  /usr/share/sysbench/oltp_read_write.lua \
  run
```

#### 4.2 观察指标（每 5 分钟记录一次）

```sql
-- 连接数变化趋势
SHOW STATUS LIKE 'Threads_connected';
SHOW STATUS LIKE 'Threads_running';

-- 锁等待与死锁
SHOW STATUS LIKE 'Innodb_row_lock_waits';
SHOW STATUS LIKE 'Innodb_row_lock_time_avg';

-- 缓冲池命中率（应 > 99%）
SHOW STATUS LIKE 'Innodb_buffer_pool_read_requests';
SHOW STATUS LIKE 'Innodb_buffer_pool_reads';

-- 慢查询数量（应保持平稳，不应持续增长）
SHOW STATUS LIKE 'Slow_queries';
```

**通过标准**：
- 全程 TPS/QPS 波动幅度 < ±15%（无明显衰减趋势）
- `Threads_connected` 峰值后恢复到初始水平（无泄漏）
- 应用进程内存无持续增长（启动后趋于稳定）
- AutoTuning 日志无 `[ROLLBACK]` 条目（无误触发回滚）
- `autotuning.sharding.hot_table_skew` ≤ 2.0（哈希分片无严重倾斜；该值 = 最热分片调用量 / 平均调用量，完全均匀时为 1.0，≤ 2.0 表示最热分片不超过平均的 2 倍）

## 七、监控与日志采集

### 7.1 MySQL 侧实时监控

```sql
-- 查看当前活跃查询（超过 1 秒的）
SELECT * FROM INFORMATION_SCHEMA.PROCESSLIST
WHERE COMMAND != 'Sleep' AND TIME > 1
ORDER BY TIME DESC;

-- 查看 InnoDB 锁等待
SELECT * FROM performance_schema.data_lock_waits;

-- 查看最近 100 条慢查询（需开启 slow_query_log）
SELECT * FROM mysql.slow_log ORDER BY start_time DESC LIMIT 100;
```

**建议开启**：
```sql
SET GLOBAL slow_query_log = ON;
SET GLOBAL long_query_time = 0.5;     -- 与 AutoTuning:SlowQueryThresholdMilliseconds 对齐
SET GLOBAL slow_query_log_file = '/var/log/mysql/slow.log';
```

### 7.2 应用侧日志采集

```bash
# 实时跟踪数据库日志
tail -f /path/to/logs/database-$(date +%Y%m%d).log | grep -E '\[SLOW\]|\[ALERT\]|\[ROLLBACK\]|\[TUNING\]'

# 压测结束后统计慢查询数量
grep '\[SLOW\]' /path/to/logs/database-*.log | wc -l

# 提取 AutoTuning TopN 慢查询报告
grep 'TopN\|P99\|sharding.hit_rate' /path/to/logs/database-*.log
```

### 7.3 分表指标观察

AutoTuning 每 30 秒输出一次分表观测指标，关注以下日志关键词：
- `autotuning.sharding.hit_rate`：命中率，应 ≥ 98%
- `autotuning.sharding.cross_table_query_ratio`：跨分表查询占比，建议 < 5%
- `autotuning.sharding.hot_table_skew`：哈希分片倾斜度（= 最热分片调用量 / 平均调用量），完全均匀时为 1.0，应 ≤ 2.0

## 八、通过/失败判定标准（验收矩阵）

| 编号 | 验收项 | 通过标准 | 失败处理 |
|------|-------|---------|---------|
| AC-01 | Scenario 1 高压 TPS | ≥ 500 TPS（32 线程） | 分析慢查询、增加索引、调整 InnoDB 参数 |
| AC-02 | Scenario 2 中压 QPS | ≥ 800 QPS（32 线程） | 检查索引覆盖、分表命中情况 |
| AC-03 | 全场景 P99 延迟 | ≤ 500 ms | 对超阈值查询执行 `EXPLAIN`，针对性优化 |
| AC-04 | 全场景 P95 延迟 | ≤ 200 ms | 同上 |
| AC-05 | 全场景错误率 | < 1% | 检查连接池配置、重试策略 |
| AC-06 | 全场景死锁次数 | = 0 | 分析死锁日志，调整事务隔离级别或业务逻辑 |
| AC-07 | 分表命中率 | ≥ 98% | 检查分表路由键是否在查询条件中 |
| AC-08 | 稳定性（30 min） | TPS 波动 < ±15%，无泄漏 | 排查连接池配置、GC 压力 |
| AC-09 | AutoTuning 误触发 | 无误触发 `[ROLLBACK]` | 复核 AutoTuning 阈值配置 |
| AC-10 | CommandTimeout | 0 次超时（30s） | 检查慢查询、加索引或拆分复杂查询 |

## 九、已知风险与规避措施

| 风险 | 描述 | 规避措施 |
|------|------|---------|
| **分表未创建** | 若目标月份分表不存在，INSERT 路由失败 | 压测前确认 `CreateShardingTableOnStarting = true` 并验证表存在 |
| **哈希分片倾斜** | 若 BagCode 分布不均匀导致部分哈希分片过热 | 种子数据使用均匀 BagCode 分布，压测后观察 `hot_table_skew` 指标 |
| **AutoTuning 干扰** | 高压期间 AutoTuning 可能自动执行 DDL（如建索引）影响 TPS | 压测期间将 `DryRun = true`，防止 DDL 影响基准数据 |
| **连接池耗尽** | 64 线程以上可能耗尽 DbContext 连接池 | 监控 `Threads_connected`，必要时增大 `max_connections` |
| **数据膨胀影响** | 百万级种子数据对缓冲池命中率有影响 | 压测前执行 `FLUSH STATUS` 重置计数器，消除历史数据影响 |
| **单机资源竞争** | 压测机与数据库同机运行时 CPU 竞争影响结果 | 建议分离压测执行机与数据库节点，或隔离 CPU 核心 |

## 十一、后续优化建议（待压测验证后填写）

1. **索引优化**：若 Q3（ScannedTime 范围查询）P99 超过 200 ms，考虑添加复合索引 `(WorkstationName, ScannedTime, Status)`。
2. **连接池调优**：若峰值连接数超过 `max_connections × 60%`，评估 DbContext 池大小与 MySQL `max_connections` 的配比。
3. **哈希分片扩容**：若 `hot_table_skew > 2.0` 且单分片行数接近 200 万（`HotLayeringRows` 阈值），触发 16→32 扩容计划（参见 `appsettings.json` `HashSharding:ExpansionPlan`）。
4. **读写分离**：若读取 QPS 成为瓶颈，可引入 MySQL 只读副本，通过连接字符串路由区分读写。
5. **归档策略**：若分表行数持续增长超过 `CapacityPrediction:HotLayeringRows = 200000`，触发冷热分层，将历史月份分表迁入归档存储。
