# 数据库底座门禁说明

## 文档目标

本文说明《Zeye.Sorting.Hub-长期数据库底座多PR实施方案与Copilot严格门禁.md》中 PR-F“数据库底座 CI 门禁增强”的当前落地方式，便于后续 PR 在不破坏数据库底座前提下继续推进。

## 当前门禁组成

### 1. `.github/workflows/database-foundation-gates.yml`

- 负责在 PR / 手动触发时执行数据库底座门禁。
- 当前拆分为“构建与测试”和“规则门禁”两个作业。
- 构建与测试固定执行：
  - `dotnet restore Zeye.Sorting.Hub.sln`
  - `dotnet build Zeye.Sorting.Hub.sln --no-restore -c Release`
  - `dotnet test Zeye.Sorting.Hub.sln --no-build -c Release`

### 2. `.github/scripts/validate-no-utc.sh`

- 检查新增或修改代码中是否出现 UTC 相关 API。
- 检查新增或修改 `appsettings*.json` 是否出现 `Z` 或 offset 时间示例。

### 3. `.github/scripts/validate-readme-file-tree.sh`

- 面向新增/删除文件校验 `README.md` 是否同步更新职责说明。
- 重点约束“新增文件未登记、删除文件未移除”两类回归。

### 4. `.github/scripts/validate-sensitive-config.sh`

- 面向新增或修改文本文件拦截高风险连接串片段与密钥字段。
- 允许 `<占位符>`、`${{ secrets.xxx }}`、`${ENV_VAR}`、`%ENV_VAR%` 这类占位写法。

### 5. `.github/scripts/validate-no-shadow-code.sh`

- 面向新增或修改的 C# 文件检查同文件重复方法签名。
- 用于阻断明显的复制粘贴式影分身代码回归。

### 6. `.github/scripts/validate-database-foundation-rules.sh`

- 作为数据库底座门禁主脚本。
- 除可驱动以上单项检查外，还补充以下结构性门禁：
  - 新增或修改枚举项必须带 `[Description("中文说明")]`
  - 新增或修改 HostedService 必须显式捕获异常
  - 新增或修改后台循环必须消费 `CancellationToken`
  - 新增或修改批量写入 / 后台队列实现必须体现有界容量

## 本地执行方式

在仓库根目录执行：

```bash
./.github/scripts/validate-no-utc.sh
./.github/scripts/validate-readme-file-tree.sh
./.github/scripts/validate-sensitive-config.sh
./.github/scripts/validate-no-shadow-code.sh
./.github/scripts/validate-database-foundation-rules.sh --check advanced
```

或直接执行完整门禁：

```bash
./.github/scripts/validate-database-foundation-rules.sh
```

## 当前边界

1. 当前门禁默认基于“PR 相对 base 分支 diff”或“本地未提交 diff”做增量扫描，优先拦截新增回归，不强制一次性清理历史遗留项。
2. `stability-gates.yml` 与 `copilot-instructions-validation.yml` 继续保留，分别承担长期稳定性门禁与 Copilot 规则快照校验。
3. 敏感配置门禁当前以文本模式识别高风险片段；若后续需要更强精度，可再演进到 AST / JSON 语义级校验。

## 下一阶段入口

- 当前已进入 PR-H 之后阶段；下一 PR 从“慢查询指纹聚合与查询画像”（PR-I）开始。
- 后续若需要扩展数据库底座门禁，应优先在 `validate-database-foundation-rules.sh` 内集中扩展，避免新增重复脚本实现。
